✅ Approval Workflow

Admin reviews and approves/rejects orders with automatic label generation and notifications

Approval Queue

Admin sees all pending orders in a dedicated approval queue with full order details for review.

Approval Workflow:

1
Salesperson creates order

Order enters PENDING_ADMIN_APPROVAL status, appears in admin's approval queue

2
Admin reviews order

Check customer details, products, pricing, payment proof (if prepaid)

3
Admin approves or rejects

Approve: Order moves to APPROVED → Label auto-generates → Notifications sent
Reject: Order marked REJECTED, salesperson notified

Auto Label Generation

🏷️ What happens on approval:

• PDF label auto-generates with Code128 barcode of order code
• Label includes customer name, address, items, COD amount
• Stored in system, printable by admin & packer
• Order status advances to LABEL_GENERATED

Note: Label PDFs are auto-generated with Code128 barcodes showing order code, customer details, items, and COD amount. PDF generation requires server-side rendering which cannot be captured in automated screenshots.

📋 Packing & Dispatch

Barcode-driven packing workflow with work queues, label printing, and handover/dispatch tracking

Packing Work Queues

Packing users see three queues organized by status:

📦 Orders to Pack

LABEL_GENERATED orders, oldest first. Packer prints label, packs items, scans barcode to mark PACKED.

🤝 Awaiting Handover

PACKED orders. Packer clicks "Handover" to mark HANDED_TO_DELIVERY when given to courier.

🚚 Awaiting Dispatch

HANDED_TO_DELIVERY orders. Admin/Packer clicks "Dispatch" when courier leaves with shipment.

Barcode Scanning

Packing Workflow:

1
Print Label

Click "Print label" button on order card → PDF opens with barcode

2
Pack Items

Physically pack items matching the order, affix printed label

3
Scan Barcode

Use scanner box or type order code, hit "Scan" → Order marked PACKED

4
Session Summary

"Ready to move" counter + recent scans list shows progress

Handover & Dispatch

Two-step fulfilment: Handover = physically given to courier (HANDED_TO_DELIVERY). Dispatch = courier leaves with shipment (DISPATCHED). Keeps accountability clear.

💰 COD Reconciliation

Track COD receivables, settle payments, handle claims for lost/damaged shipments

Receivables Management

All COD orders create receivables. Admins/Accountants track outstanding amounts and settle when courier remits.

📋 Outstanding View

All unsettled receivables grouped by courier, filterable by status and date range.

✅ Settlement

Mark as SETTLED with amount received, UTR/reference, settlement date. Tracks partial payments.

📦 RTO Handling

Orders marked RTO create loss entries. Track returned stock and courier charges.

⚠️ Lost/Damaged Claims

For COURIER_LOST orders, file claims with claim amount, courier response, resolution.

Settlement Workflow

1
COD Order Delivered

System creates a receivable (PENDING) with amount = order COD total

2
Courier Remits Payment

Accountant receives bank transfer or cheque from courier company

3
Settle in System

Find receivable, click "Settle", enter amount, UTR, date → Status = SETTLED

4
Reports & Audit

Settled receivables appear in finance reports and audit log

Note: The "Settle Receivable" modal appears when clicking "Settle" on pending COD amounts. This modal interaction requires manual demonstration and is not captured in automated screenshots. The receivables list view is shown above.

Claims Management

Lost/Damaged Shipments: File claims directly in the system with claim amount, supporting documents, and courier response tracking. Links to the original order for full context.

Profit & Loss (P&L)

Monthly P&L report showing revenue, expenses, and net profit/loss.

📈 P&L Calculation

Revenue: Sum of all DELIVERED orders (excluding REJECTED/CANCELLED)
Expenses: Sum of all tracked expenses
Net Profit/Loss: Revenue - Expenses

Export to Vyapar

Accounting Integration: Export order data in Vyapar-compatible CSV format for seamless accounting software sync.

👤 Staff Management NEW

Complete staff profiles, ID verification, photo uploads, and self-service profile change requests

Salespeople Directory (Admin Only)

Admins see all salespeople with full profiles, verification status, and document uploads.

📋 Profile Details

  • Full name, email, mobile
  • Date of birth, address
  • Joined date
  • Role assignment

🖼️ Two Images

  • Profile photo (max 600px, compressed)
  • Government ID photo (max 1600px, compressed)
  • Both stored in S3/DB with auto-compression

🆔 ID Verification

  • ID type: Aadhaar, PAN, Driving License, Voter ID, Passport
  • ID number captured
  • Status: PENDING, VERIFIED, REJECTED

ID Verification Workflow

1
Salesperson Created

Admin creates user → Status defaults to PENDING (from V31 migration)

2
Admin Uploads Documents

Profile photo + Govt ID photo uploaded → Compressed server-side → Status stays PENDING

3
Admin Verifies

Review docs → Click "Verify" or "Reject" with optional note → Status = VERIFIED/REJECTED, audited

My Profile (Self-Service)

All staff can view their own profile and request changes that require admin approval.

Self-Service Features:

  • View own profile & verification status
  • View profile photo (upload/change is admin-only by design)
  • Submit change request for: name, email, mobile, DOB, address, ID details
  • See pending request status with current vs. proposed diff

Profile Change Approval (Admin Only)

Admins review and approve/reject staff profile change requests.

🔐 Moderated Changes

Staff cannot edit their core details directly. Instead, they submit a change request → Admin sees current vs. proposed side-by-side → Approve applies changes to user record OR Reject leaves unchanged. This keeps accountability intact.

🔒 Security & Audit

JWT authentication, role-based access control, and comprehensive audit logging

Authentication

Secure JWT-based login with password hashing (BCrypt) and session management.

🔐 Security Features

  • BCrypt password hashing (strength 12)
  • JWT tokens (24h expiry)
  • Refresh token mechanism
  • Auto logout on token expiry

🚫 Protection

  • CORS configured for allowed origins
  • CSRF protection
  • Rate limiting on sensitive endpoints
  • SQL injection prevention (JPA/Hibernate)

Role-Based Access Control (RBAC)

Five distinct roles with granular permissions:

RoleKey PermissionsCannot Do
ADMIN Everything: approve orders, manage users, settings, reports, insights —
SALESPERSON Create orders, manage own leads, view own orders, read products/customers Cannot approve orders, edit products, see other salespeople's data
PACKING_USER View packing queues, scan barcodes, print labels, handover/dispatch Cannot create orders, see financials, approve anything
ACCOUNTANT Reconcile COD, settle receivables, manage expenses, view P&L Cannot approve orders, create orders, manage products
CUSTOMER (Future: storefront login, order history) No admin panel access in current dashboard-only pivot

Audit Trail

Every sensitive action is logged with who, what, when, and context.

Audited Actions:

Orders

Created, Approved, Rejected, Status Changed, Dispatched

Users

Created, Activated, Deactivated, Password Reset, Profile Updated

Finance

Receivable Settled, Expense Added, Invoice Generated

Settings

Settings Updated, Courier Added, Product Created/Updated

Audit Log Fields: Timestamp, User (who), Action (what), Entity Type (order/user/etc.), Entity ID, Details (JSON context). Filterable by date, user, action, entity. Exportable for compliance.
